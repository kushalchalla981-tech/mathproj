# Technical Requirement Document

**Product:** PCA based Image Compression and Orientation Correction
**Version:** V1 Architecture

---

# 1. System Architecture Overview

The system uses a modular architecture. It separates compute intensive image processing from the interface layer. The goal is predictable performance, easy testing, and long term maintainability.

### Core Architecture Layers

**Client Layer**

* CLI client
* GUI application
* External applications using library API

**Service Layer**

* Compression service wrapper
* Orientation detection module
* Quality metrics module

**Core Compute Layer**

* PCA compression engine
* Eigen decomposition module
* Tile processing engine

**Utility Layer**

* Image loader and decoder
* Metadata processor
* Logging and metrics

### Deployment Modes

Three usage modes share the same compute engine.

1. **Local CLI**

   * Single binary
   * No server required

2. **Local GUI Application**

   * Desktop app calling the local library

3. **Optional Local HTTP Service**

   * For pipelines and integrations

### Logical Flow

1. Image is loaded.
2. Metadata and EXIF are read.
3. Image converted to RGB if needed.
4. Image divided into tiles if large.
5. Mean centering performed.
6. Covariance matrix calculated.
7. Eigen decomposition executed.
8. Dominant eigenvector projection applied.
9. Image reconstructed.
10. Orientation corrected.
11. SSIM or PSNR calculated.
12. Output encoded and returned.

# 2. Frontend Responsibilities

Frontend includes both **CLI interface** and **GUI application**.

Frontend must remain lightweight. No heavy computation runs here.

### CLI Responsibilities

Input parsing.

Supported flags:

* input path
* output path
* compression strength
* PCA mode
* tile size
* orientation override
* EXIF handling
* batch folder mode

Execution flow:

1. Validate user input.
2. Load file or folder.
3. Send request to compression library.
4. Display results.

CLI output:

* compression ratio
* SSIM or PSNR
* processing time
* output file path

Batch mode output:

* CSV report

---

### GUI Responsibilities

GUI acts as a visual test interface.

Key functions:

Image preview
Compression slider
Export button
Metrics display

Workflow:

1. User selects image.
2. GUI sends image to backend.
3. Backend returns compressed preview.
4. GUI displays side by side comparison.

Preview refresh must be under **500 ms** for usability.

GUI components:

* Image viewer
* Compression slider
* Export panel
* Metrics display

---

# 3. Backend Responsibilities

Backend contains all processing logic.

### Core Modules

**1. Image Loader**

Responsibilities:

* Decode JPEG and PNG
* Extract EXIF
* Convert color spaces
* Normalize pixel arrays

Supported formats:

* JPEG
* PNG

---

**2. Preprocessing Module**

Operations:

* Mean centering
* Tile segmentation
* Color channel splitting

Output:

* Mean centered pixel matrices

---

**3. PCA Compression Engine**

Steps:

1. Compute covariance matrix
2. Perform eigen decomposition
3. Identify dominant eigenvector
4. Project pixel data
5. Reconstruct reduced representation

Supports:

* Per channel PCA
* Joint RGB PCA

---

**3b. Eigen Analysis Module**

Purpose:

Provide eigenvalue and eigenvector analysis for display and visualization.

Functions:

* analyze_image() - Perform PCA eigen-decomposition on image
* get_axis_overlay() - Generate overlay coordinates for visualization
* format_eigenvalue_scientific() - Format values in scientific notation

Data Structures:

* EigenAnalysisResult
  - eigenvalues: Vec<f64> (λ₁, λ₂, λ₃)
  - eigenvectors: Vec<f64> (3x3 matrix)
  - variance_explained: Vec<f64> (percentages)
  - cumulative_variance: Vec<f64>
  - principal_axis_angle: f64 (degrees)
  - confidence: f64 (0.0-1.0)
  - recommended_rotation: f32 (0, 90, 180, 270)

* AxisOverlay
  - x1, y1, x2, y2: f64 (normalized 0-1)
  - angle: f64
  - primary_eigenvalue: f64

* OverlayColor
  - Red, Yellow, Cyan, Green, Magenta

---

**4. Tile Processing Engine**

Purpose:

Prevent memory spikes.

Process:

1. Split image into tiles
2. Apply PCA per tile
3. Reconstruct tiles
4. Stitch results

Tile size default:

1024 x 1024

---

**5. Orientation Correction Engine**

Inputs:

* Principal axis
* EXIF orientation

Logic:

1. Calculate principal orientation vector.
2. Compare with EXIF orientation.
3. If variance confidence low, fallback to EXIF.

---

**6. Quality Metric Module**

Supported metrics:

* SSIM
* PSNR

Purpose:

Evaluate visual similarity between original and reconstructed images.

---

**7. Encoding Module**

Encodes final output.

Supported formats:

* JPEG
* PNG

Compression level controlled by quality parameter.

---

**8. Logging Module**

Records:

* processing time
* input size
* output size
* compression ratio
* SSIM

Export formats:

* CSV
* structured logs

---

# 5. API Structure

API exists only for service integration.

Base path:

```
/v1
```

---

### Compress Image

```
POST /v1/compress
```

Request:

```
multipart/form-data
```

Parameters:

| Field             | Type   |
| ----------------- | ------ |
| file              | binary |
| mode              | string |
| quality           | float  |
| retain_components | int    |
| orientation       | string |
| tile_size         | int    |

Response:

```
{
  "compressed_image": bytes,
  "size_before": int,
  "size_after": int,
  "compression_ratio": float,
  "ssim": float,
  "orientation_used": "pca"
}
```

---

### Batch Compression

```
POST /v1/compress/batch
```

Input:

Folder upload or list of images.

Response:

Job ID.

---

### Batch Job Status

```
GET /v1/batch/{job_id}
```

Returns:

* processed count
* failures
* report file

---

# 8. Scalability Considerations

System must handle both single images and batch workloads.

### Memory Control

Tile based processing prevents loading full image matrices.

User configurable limit:

```
--max-ram
```

---

### CPU Parallelism

Batch jobs run across worker threads.

Thread model:

```
worker_pool = CPU cores - 1
```

Each worker processes one image.

---

### Streaming Covariance

For extremely large images:

* incremental covariance calculation
* avoids full matrix storage

---

### Horizontal Scaling

If deployed as API service:

Scaling strategy:

Stateless workers.

Architecture:

```
Load Balancer
      |
Compression Workers
```

Workers store temporary files locally.

Results returned immediately.

No shared state required.

---

### GPU Acceleration (Future)

Potential integration:

* CUDA
* OpenCL

Operations suitable for GPU:

* matrix multiplication
* covariance calculation

Not required for V1.
